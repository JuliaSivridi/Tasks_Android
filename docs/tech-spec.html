1. Overview

Stler Tasks is a native Android task manager backed by Google Sheets as the database. There is no dedicated backend server — all persistent data lives in a Google Spreadsheet (db_tasks) accessed via the Sheets API v4. Room acts as a local cache, making the app fully functional offline. All UI is built in Jetpack Compose.

Key Design Goals

  • Google Sheets as the single source of truth — the same spreadsheet is shared with the PWA
  • Offline-first — full read/write via Room; a sync queue flushes changes when online
  • Reactive UI — all screens observe Room Flows; data updates propagate automatically without manual refresh
  • Clean MVVM + Repository with Hilt DI throughout
  • Glance widgets that react to Room changes without explicit updateAll() calls
  • Zero onboarding friction — the db_tasks spreadsheet is created automatically on first sign-in

3. Architecture

Pattern: MVVM + Repository + Unidirectional Data Flow

UI (Composable)
    ↕  StateFlow / collectAsStateWithLifecycle
ViewModel  (Hilt, ViewModelScope)
    ↕  suspend functions / Flow
Repository  (Singleton, @Inject)
    ↕  Room DAOs (Flow)          ↕  SyncQueue (write path)
Local DB (Room)                  SyncWorker (background)
                                     ↕  Retrofit / OkHttp
                                 Google Sheets API v4

Write path: Every mutation → Room write (instant, triggers recomposition) + SyncQueue entry. SyncWorker drains the queue on next sync.

Read path: Room Flow → ViewModel StateFlow → Composable. Sheets data is only read during the sync pull (fetchAllAndSave), which upserts into Room.

Error Handling

BaseViewModel.safeLaunch wraps all viewModelScope.launch calls. CancellationException is always rethrown. All other exceptions are emitted to a Channel<String> (uiError Flow). Each screen subscribes via ErrorSnackbarEffect and shows a Snackbar. A LocalSnackbarHostState CompositionLocal and a single SnackbarHost in MainScreen handle display.

7. Authentication & First-Launch Setup

7.1 OAuth Scopes

  • https://www.googleapis.com/auth/spreadsheets — full Sheets read/write + create new spreadsheets
  • https://www.googleapis.com/auth/drive.metadata.readonly — search Drive for existing spreadsheet by name

7.2 Sign-In Flow

  1. CredentialManager → Google account picker → GoogleIdTokenCredential with email, display name, avatar URL
  2. Identity.getAuthorizationClient.authorize() → requests both scopes → may return a PendingIntent if user hasn't approved yet
  3. If hasResolution() → return SignInStep.NeedsAuthorization → AuthScreen launches the intent → finalizeAuth() is called on result
  4. findOrCreateSpreadsheet(token) → Drive API search → if not found, create (see §7.3)
  5. Save to DataStore: accessToken, tokenExpiry (+1 h), spreadsheetId, userEmail, userName, userAvatarUrl

7.3 First-Launch Spreadsheet Creation

When no db_tasks spreadsheet exists in the user's Drive, createSpreadsheet() runs automatically:

  1. POST https://sheets.googleapis.com/v4/spreadsheets — creates spreadsheet with 3 named sheets: tasks, folders, labels
  2. POST .../values:batchUpdate — writes header row to each sheet + seeds the Inbox folder row (fld-inbox, Inbox, #6b7280, 0)
  3. Room upsert — seeds FolderEntity(id="fld-inbox") locally so the app is immediately usable before first sync
Both API calls use a plain OkHttpClient (not Retrofit) to avoid a circular dependency with the Hilt-managed client that requires a valid TokenProvider.

7.4 Token Refresh

refreshToken() calls Identity.getAuthorizationClient.authorize() silently — no Activity needed. Token is considered stale if it expires within 5 minutes. The NetworkModule 401 Authenticator calls refreshToken() automatically on HTTP 401 responses.

7.5 Sign-Out

Clears all DataStore keys + deletes all Room tables (tasks, folders, labels, sync_queue).

8. Synchronization

8.1 SyncWorker — Execution Order

  1. Push — drain SyncQueue ordered by insertion:
    • INSERT → SheetsApi.append(range = "sheet!A:lastCol")
    • UPDATE → fetch current rows (cached per sheet per push) → find row number → SheetsApi.batchUpdate
    • DELETE → same row-number lookup → SheetsApi.clear
    • On success: syncQueueDao.deleteById
    • On failure: syncQueueDao.incrementRetry
    • After all items: deleteExhausted() removes items with retryCount ≥ 5
  2. Delay 1 s (only if there were pending items) — lets Sheets process the write before the read, preventing stale-cache overwrite of freshly pushed data
  3. Pull — taskRepository.fetchAllAndSave(spreadsheetId):
    • SheetsApi.batchGet(["tasks", "folders", "labels"])
    • All three entities upserted inside a single Room transaction (db.withTransaction)

WorkManager retry: up to 4 attempts with exponential backoff; returns Result.failure() after the 5th.

8.2 SyncManager

  • Periodic: PeriodicWorkRequest every 30 min, constraint NetworkType.CONNECTED, unique name with KEEP policy
  • One-off: triggerSync() enqueues an immediate OneTimeWorkRequest, replaces existing with KEEP policy
  • Initialized in TasksApplication.onCreate()

8.3 SyncState

StateFlow<SyncState> combining WorkManager WorkInfo + SyncQueueDao.observePendingCount():

  • Idle — no running worker, queue empty
  • Syncing — worker is RUNNING
  • Pending(count) — queue has items, no worker running

Displayed in the sidebar footer: count badge + spinning icon when Syncing.

8.4 Widget Refresh After Mutations

WidgetRefresher is injected into TaskRepositoryImpl and called after every mutation (complete, update, delete, reorder, restore). It uses a debounced (400 ms) MutableSharedFlow trigger to call updateAll() on all three widget types without flooding Glance with redundant update requests.

16. CI/CD & Build

Workflow: .github/workflows/release.yml
Trigger: Push of a tag matching v* (e.g. v1.3)

Pipeline Steps

  1. Checkout code (actions/checkout@v4)
  2. Set up JDK 17 (actions/setup-java@v4)
  3. Make gradlew executable
  4. Decode keystore from KEYSTORE_BASE64 secret → write to temp file
  5. Set env vars: KEYSTORE_PATH, KEYSTORE_PASSWORD, KEY_ALIAS, KEY_PASSWORD
  6. Run ./gradlew assembleRelease
  7. Rename APK to stler-tasks.apk
  8. Create GitHub Release with APK as artifact (softprops/action-gh-release@v2)

Signing Config

app/build.gradle.kts reads KEYSTORE_PATH env var. The signing config block is only created when the var is non-blank — safe for local debug builds without a keystore. The release build type only applies signing when the config exists (signingConfigs.findByName("release")).

Minification: isMinifyEnabled = false. ProGuard rules file exists but is not currently applied.

17. First-Time Setup (New Developer)

  1. Clone the repository from github.com/JuliaSivridi/Tasks_Android
  2. In Google Cloud Console:
    • Enable Google Sheets API and Google Drive API
    • Create OAuth 2.0 Web Client ID → copy to app/src/main/res/values/strings.xml as google_web_client_id
    • Create OAuth 2.0 Android Client ID (package com.stler.tasks, SHA-1 of your debug keystore)
    • Set OAuth consent screen to Production (so any Google account can sign in, not just test users)
  3. Run on a device or emulator from Android Studio
  4. Sign in — the app will automatically find or create the db_tasks spreadsheet

For release builds: Create a keystore, base64-encode it, add GitHub secrets KEYSTORE_BASE64, KEYSTORE_PASSWORD, KEY_ALIAS, KEY_PASSWORD. Push a v* tag to trigger the CI build.

Distribution: Users download the signed APK from the GitHub Release, enable "Install from unknown sources" in Android settings, and install directly — no Play Store involvement required.

18. Key Algorithms

18.1 Recurring Task Completion

if (task.isRecurring) {
    val newDate = when (task.recurType) {
        DAILY   → deadlineDate.plusDays(recurValue)
        WEEKLY  → deadlineDate.plusWeeks(recurValue)
        MONTHLY → deadlineDate.plusMonths(recurValue)
    }
    // Update deadlineDate only.
    // status stays PENDING, completedAt = "", updatedAt = now.
} else {
    // Set status = COMPLETED, completedAt = now.
    // Recursively complete all descendants (all tasks loaded once,
    // then traversed in memory — no N+1 queries).
}

18.2 FolderScreen Tree Flattening

// FolderViewModel builds List<DisplayNode>:
1. Build map: parentId → children (PENDING tasks only, sorted by sortOrder)
2. DFS from root tasks (parentId == ""), respecting task.isExpanded
3. Each DisplayNode: task, depth, childCount, completedChildCount
4. localList = this flattened list (mutable copy for optimistic reorder UI)

18.3 Drag-to-Reorder (FolderScreen)

// pendingReorder = plain object (NOT mutableStateOf — avoids recomposition on every drag frame)
// Stores: parentId, fromIdx (in original displayList), toIdx (in localList)

onMove { from, to →
    // 1. Update localList optimistically (pure UI, no DB)
    localList = localList.toMutableList().apply {
        add(to.index, removeAt(from.index))
    }
    // 2. Store final sibling indices
    pendingReorder.fromIdx = siblings.indexOfFirst { it.task.id == draggedNode.task.id }
    pendingReorder.toIdx   = siblingsInLocal.indexOfFirst { it.task.id == draggedNode.task.id }
}

LaunchedEffect(isDragging) {
    if (!isDragging && pendingReorder indices are valid and different) {
        viewModel.reorderSiblings(parentId, fromIdx, toIdx)
        // → single batch Room upsertAll + one SyncQueue entry
        pendingReorder.fromIdx = -1  // reset
    }
}

18.4 Swipe-to-Complete Key Invalidation

// key(task.id, task.deadlineDate) { ... } wraps:
//   rememberSwipeToDismissBoxState + LaunchedEffect + SwipeToDismissBox
//
// When deadline changes (recurring task completed) →
//   key changes → Compose tears down and rebuilds inner state →
//   rememberSaveable resets to Settled →
//   swipe works again immediately (no snapTo needed)
//
// When navigating back to a screen →
//   rememberSaveable would normally restore StartToEnd state →
//   key(task.deadlineDate) invalidates the saved state →
//   no spurious re-completion of recurring tasks
