1. Overview

Stler Tasks is a personal task manager for Android. It is a native rewrite of a PWA with the same Google Sheets backend, so both apps share one db_tasks spreadsheet per user. There is no dedicated backend server — all persistent task data lives in Google Sheets, accessed via the Sheets API v4. Room serves as a local cache that makes the app fully functional offline.

In addition to tasks, the app integrates with Google Calendar API v3: events from selected calendars are fetched, cached in Room, and displayed alongside tasks in the Upcoming screen, Calendar screen, and all widgets.

Key design goals:

  • Google Sheets as the single source of truth for tasks — no proprietary cloud service
  • Full offline support via Room + sync queue
  • Reactive UI — all screens observe Room Flows; data updates propagate automatically
  • Clean MVVM + Repository architecture with Hilt DI throughout
  • Glance widgets that react to Room changes without explicit refresh calls
  • Google Calendar events surfaced inline with tasks, unified by date

3. Architecture

Pattern: MVVM + Repository + Unidirectional Data Flow

UI (Composable)
    ↕ StateFlow / collectAsStateWithLifecycle
ViewModel (Hilt, ViewModelScope)
    ↕ suspend / Flow
Repository (Singleton)
    ↕ Room DAOs (Flow)     ↕ SyncQueue (write path)
Local DB (Room)            SyncWorker (background push + pull)
                               ↕ Retrofit / OkHttp
                           Google Sheets API v4
                           Google Calendar API v3

Write path (tasks): Every mutation → Room write (instant, triggers UI recomposition) + SyncQueue INSERT. SyncWorker drains the queue on next sync.

Write path (calendar events): Mutations (create / update / delete) go directly to the Calendar API via CalendarRepository. On success, Room is updated in place — no SyncQueue, no Google Sheets involvement.

Read path: Room Flow → ViewModel StateFlow → Composable. Sheets data is only read during sync pull (fetchAllAndSave); Calendar events are fetched by SyncWorker after the Sheets pull step.

Error handling: BaseViewModel.safeLaunch wraps all viewModelScope.launch calls, catches exceptions, and emits to a Channel<String> (uiError Flow). ErrorSnackbarEffect in each screen subscribes and shows a Snackbar.

6. Authentication & First-Launch Setup

6.1 OAuth Scopes

  • https://www.googleapis.com/auth/spreadsheets — full Sheets read/write + create
  • https://www.googleapis.com/auth/drive.metadata.readonly — search Drive for existing spreadsheet
  • https://www.googleapis.com/auth/calendar — read/write Google Calendar events

6.2 Sign-In Flow

  1. CredentialManager → shows Google account picker → returns GoogleIdTokenCredential with user email, display name, avatar URL
  2. Identity.getAuthorizationClient → requests Sheets + Drive + Calendar scopes → may return a PendingIntent if user hasn't approved yet
  3. If hasResolution() → return SignInStep.NeedsAuthorization(pendingIntent) → AuthScreen launches the intent → finalizeAuth() called on result
  4. findOrCreateSpreadsheet(token) → Drive API search for name='db_tasks' → if not found, creates new spreadsheet
  5. Save to DataStore: accessToken, tokenExpiry (+1 h), spreadsheetId, userEmail, userName, userAvatarUrl

6.3 First-Launch Spreadsheet Creation

  1. POST https://sheets.googleapis.com/v4/spreadsheets — creates spreadsheet with 3 named sheets (tasks, folders, labels)
  2. POST .../values:batchUpdate — writes header rows + seeds Inbox folder row
  3. Upserts FolderEntity(id="fld-inbox", ...) to Room so the app is immediately usable

6.4 Token Refresh

refreshToken() calls Identity.getAuthorizationClient.authorize() silently. Token is stale if it expires within 5 minutes. The NetworkModule 401 Authenticator calls refreshToken() automatically.

6.5 Sign-Out

Clears all DataStore keys + deletes all Room tables (tasks, folders, labels, sync_queue, calendar_events).

7. Synchronization

7.1 SyncWorker (push → pull Sheets → pull Calendar)

  1. Push — drain SyncQueue ordered by insertion:
    • INSERT → SheetsApi.append
    • UPDATE → row-number lookup → SheetsApi.batchUpdate
    • DELETE → row-number lookup → SheetsApi.clear
    • On success: delete queue item. On failure: increment retry. After all: remove exhausted (retryCount ≥ 5)
  2. Delay — 1 000 ms if there were pending items (lets Sheets process writes)
  3. Pull Sheets — SheetsApi.batchGet(["tasks","folders","labels"]) → upsert all three in a single Room transaction
  4. Pull Calendar — for each selected calendar: CalendarApi.listEvents(from=today, to=today+30d) → upsert to calendar_events, delete stale events

Retry: up to 4 attempts (WorkManager exponential backoff).

7.2 Calendar Event Write Path

Calendar mutations bypass the SyncQueue entirely and go directly to the Calendar API:

OperationAPI methodNotes
CreatePOST eventsSaves new event to Room on success
UpdatePUT events/{id}Replaces entire event — avoids residual start.date from PATCH merge
DeleteDELETE events/{id}Removes from Room on success
Move calendarPOST events/{id}/moveCalled before update when calendar changed
EventDateTime serialization: A custom Gson TypeAdapter (EventDateTimeAdapter) omits null fields by default (serializeNulls=false), ensuring only date or only dateTime is sent — never both. The timeZone field (IANA name) is required by the Calendar API for recurring timed events.

7.3 SyncManager

  • Periodic: PeriodicWorkRequest every 30 min, constraint CONNECTED
  • One-off: triggerSync() enqueues an immediate OneTimeWorkRequest, replaces existing with KEEP policy

7.4 SyncState

StateFlow<SyncState> combining WorkManager WorkInfo + SyncQueueDao.countAll(): Idle / Syncing / Pending(count). Displayed in the sidebar footer.

15. CI/CD

Workflow: .github/workflows/release.yml  ·  Trigger: push of a tag matching v* (e.g. v1.1)

  1. Checkout code
  2. Set up JDK 11
  3. Decode keystore from KEYSTORE_BASE64 secret → write to temp file
  4. Set env vars: KEYSTORE_PATH, KEYSTORE_PASSWORD, KEY_ALIAS, KEY_PASSWORD
  5. ./gradlew assembleRelease
  6. Upload signed APK as GitHub Release artifact

app/build.gradle.kts reads KEYSTORE_PATH env var; signing config is only created when the var is set (safe for local debug builds).

16. First-Time Setup (New Developer)

  1. Clone the repository
  2. In Google Cloud Console:
    • Enable Google Sheets API, Google Drive API, Google Calendar API
    • Create OAuth 2.0 Web Client ID → copy to res/values/strings.xml as google_web_client_id
    • Create OAuth 2.0 Android Client ID (package com.stler.tasks, SHA-1 of debug keystore)
    • Set OAuth consent screen to Production and add all three scopes
  3. Run on device/emulator from Android Studio
  4. Sign in — the app will find or create the db_tasks spreadsheet automatically

For release builds: Create a keystore, base64-encode it, add GitHub secrets KEYSTORE_BASE64, KEYSTORE_PASSWORD, KEY_ALIAS, KEY_PASSWORD. Push a v* tag to trigger the build.

17. Key Algorithms

Recurring Task Completion

if (task.isRecurring) {
    val newDate = when (task.recurType) {
        DAILY   -> LocalDate.parse(deadlineDate).plusDays(recurValue.toLong())
        WEEKLY  -> LocalDate.parse(deadlineDate).plusWeeks(recurValue.toLong())
        MONTHLY -> LocalDate.parse(deadlineDate).plusMonths(recurValue.toLong())
    }
    // update deadlineDate only; status stays PENDING; completedAt = ""
} else {
    // set status = COMPLETED, completedAt = now
    // recursively complete all descendants
}

FolderScreen Tree Flattening

FolderViewModel builds a List<DisplayNode>:

  1. Build map parentId → children (PENDING only)
  2. DFS from root tasks (parentId == "") respecting isExpanded
  3. Each node carries depth, childCount, completedChildCount
  4. localList (optimistic reorder) updated on drag; DB write only on drop

Upcoming / Widget Timeline Merge

Tasks and events merged into TimelineEntry(date, hasTime, time, row) and sorted:

compareBy({ it.date }, { if (it.hasTime) 0 else 1 }, { it.time })

Grouped by date with a Header row inserted before each group. Overdue entries (date < today) collected into a single "Overdue" group at the top.

EventDateTime Serialization

Calendar API requires exactly one of date or dateTime per event boundary:

All-day:  EventDateTime(date = "YYYY-MM-DD")
Timed:    EventDateTime(dateTime = "2026-05-04T14:00:00+03:00",
                        timeZone = "Europe/Helsinki")

Custom EventDateTimeAdapter serializes with serializeNulls=false — absent fields are omitted from JSON. timeZone is required by Google Calendar API for recurring timed events.

RRULE Builder (CustomRecurrenceSheet)

RRULE:FREQ=WEEKLY;INTERVAL=2;BYDAY=MO,WE,FR;COUNT=10
RRULE:FREQ=MONTHLY;INTERVAL=1;BYDAY=2TU        // 2nd Tuesday
RRULE:FREQ=MONTHLY;INTERVAL=1;BYDAY=-1SA       // last Saturday
RRULE:FREQ=DAILY;INTERVAL=1;UNTIL=20261231T235959Z

Ordinal prefix: ceil(dayOfMonth / 7.0).toInt() for 1st–4th; -1 when the day can fall in the last position (dayOfMonth > 21).
